PB Solver GPU (torch) Port — Correct, but Speed Is Workload-Dependent

The nonlinear Poisson–Boltzmann Newton-PCG solver was translated line-for-line from NumPy to torch so it can run on the same GPU as the MACE training forward, with no host round-trip. It reproduces the NumPy result bit-for-bit and passes every cal_18 accuracy gate in float64. On speed the honest answer is nuanced: on a single high-resolution cal_18 solve the GPU is not faster than the optimized CPU solver (one-time overhead dominates), but its steady-state per-step cost is ~4.7× lower, and the training workload — a ~18–29× smaller grid solved repeatedly from a warm state — is where that per-step win is expected to compound. Same solver parameters throughout (tol 1e-3, max-outer 12, cg-max-iter 40); all runs on one gpu-a100-dev node.

Why a Single cal_18 Solve Isn't Faster on GPU

Stage-stacked wall time on the 168×168×384 (≈10.8 M point) cal_18 grid, half-spectrum, float64. The GPU shrinks the steady fixsteps 1–4 dramatically, but cal_18 runs only four of them, while the one-time cost — CPU-side setup (shared by both), the coarse warm start, fixstep 0, and torch/cuFFT initialization — dominates the wall clock. There are simply not enough steady steps for the per-step win to pay back the fixed overhead.

Read it this way: compare the green (fixsteps 1–4) segments — the GPU bars are a sliver where the CPU bar is wide. That is the real speedup. The dark one-time segments (coarse + fixstep 0 + init/I/O) are what a single solve pays once; training amortizes them across thousands of solves.

float32 diverges. The PB linear operator is ill-conditioned (the Poisson g² spectrum spans a wide dynamic range); in float32 the preconditioned CG cannot reach the 1e-3 tolerance and the outer loop hits its cap at NaN. Production must use float64. On the A100 (FP64:FP32 = 1:2) a working mixed-precision path would cap at ~2× — the same as the rfft win, but far riskier — so it is deferred.

CPU vs GPU — What Actually Differs

  • Same math, different backend. pure_python/torch_pb.py mirrors grid.py + pb.py operator-for-operator: the FFT amplitude convention, reciprocal/derivative meshes, cavity construction, ion and bound-charge densities, the scalar and tensor-field dielectric response, l_op, and the CG inner loop. Both full- and half-spectrum are supported (TorchGrid(rspec=…)), each matching NumPy bit-for-bit.
  • Where GPU wins: the steady solve. Once torch/cuFFT plans are warm, each Newton+CG fixstep is FFT- and elementwise-bound — exactly what the GPU does well. cal_18 fixsteps 1–4: 6.5 s/step (CPU) → 1.4 s/step (GPU), 4.7×.
  • Where GPU loses on a single solve: fixed overhead. ~22 s of setup (CHGCAR + reference read, solute potential, cavity) runs on the CPU for both backends; torch adds a one-time CUDA context + cuFFT plan build; and cal_18's coarse-init + fixstep 0 carry the bulk of the Newton iterations. Four steady steps can't amortize that.
  • Why training is the favorable case. Training solves a model-reconstructed density on a coarse grid (≈0.3 Å, ~0.4–0.6 M points vs cal_18's 10.8 M) repeatedly; the CUDA context initializes once for the whole run, and every structure is a warm steady solve. That is the regime the 4.7× per-step number lives in.

Honest Conclusion & Next Step

The GPU port is correct and the steady-state is faster, but net training benefit is not yet proven. The 4.7× steady-step and ~0.9 s training-grid figures are measured against torch on CPU and against cal_18's own fixsteps — not against a full training loop using the optimized NumPy+C solver. The decisive test is an A/B of two complete training paths (existing NumPy-CPU pb_solvent vs the new torch-GPU backend, device-resident, warm-started) measuring real s/step at ~0.3 Å. Until that number is in, "GPU is worth it for training" stays a well-supported expectation, not a fact. A grid-convergence study (how coarse a grid still reproduces the reference ρ_ion(z)) should fix the spacing first — a coarser grid makes the steady-state win even purer.
